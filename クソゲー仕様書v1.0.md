作成日: 2026-04-21

## 0. プロジェクト概要

### 0-1. コンセプト
小学6年生レベルのクイズを、崩壊したUIで解かせるスコアアタック型クソゲー。
SNSバズを最大目標とする一作目MVP。マネタイズなし。

### 0-2. 成功指標
- SNS（X/TikTok/YouTube Shorts）での自然拡散
- リザルト画像のスクショ拡散
- 実況動画の発生
- Sランク達成者が話題化

### 0-3. 最優先事項
**バグなしリリース > 機能性 > こだわり**
一作目なので、機能カットしてでも確実に出すことを優先する。

## 2. ゲーム全体構造

### 2-1. 基本フロー
```
タイトル画面
  ↓
ステージ選択画面（1〜10、未クリアはロック）
  ↓
問題画面（1ステージ=20問）
  ↓
ステージリザルト
  ↓
```

- 全10ステージ通し: 60〜90分（Sランク狙いは無限）

## 3. ステージ構造

### 3-1. ステージ一覧

※「クリア想定率」は **SSランク取得率** のこと。Dランクは実質誰でも出せる（＝ステージ突破は容易）ため、難度指標としては「SSを取れる割合」で定義する。

| Stage | 崩壊UI数 | 特徴 | SS取得想定率 |
|---|---|---|---|
| 1 | 4 | チュートリアル兼ウォームアップ | 95% |
| 2 | 5 | 軽め | 85% |
| 3 | 7 | ジワジワ上昇 | 70% |
| 4 | 10 | 崩壊が支配的に | 50% |
| 5 | 12 | 中間難関 | 30% |
| 6 | 15 | | 15% |
| 7 | 19 | | 7% |
| 8 | 20 | 全崩壊発動 | 3% |
| 9 | 20 + パラメータMAX | 最難関 | 1% |
| 10 | 20 + 理不尽ギミック | ガチキチ（Sランク専用） | 0.01% |

### 3-2. 1ステージの構造
- 全20問
- 問題はジャンル配分に従ってランダム抽選
- 難易度はステージに応じて上昇
- 崩壊UIはステージプールからランダム選択（リプレイ性確保）

### 3-3. ステージ進行ルール
- 各ステージクリアで次ステージ解放
- Stage9クリアでSランク挑戦権
- Stage10は「挑戦モード」として独立（何度でも再挑戦可）

## 4. Stage9（最難関）詳細

### 目的
理論上クリア可能な最高難度。Aランク/Sランク分岐の関門。

### 仕様
- 崩壊UI 20個全発動
- 崩壊の発動タイミングが動的に切り替わる（5秒おきに組み合わせ変動）
- 各崩壊パラメータをMAX値に設定
  - 反転角度: 最大
  - ぼかし強度: 最大
  - 入力遅延: 最大
  - フェードアウト速度: 最速
- 1ミスごとに次問題の崩壊UIが1個追加（成功ペナルティ）
- 20問中15問以上正解でクリア

---

## 5. Stage10（ガチキチ）詳細

### 目的
クリア不能レベルの理不尽ゾーン。達成者の発生自体をバズ素材化する。

### 特殊ギミック（Stage10限定）

| # | ギミック | 仕様 |
|---|---|---|
| G1 | ランダム即死 | 問題表示前に10%の確率で即失格 |
| G2 | 誤判定 | 正解しても15%の確率で不正解扱い |
| G3 | 遅延失格 | タップが遅すぎても失格 |
| G4 | 文字化け問題 | 50%の問題で文字が記号化（推測ゲー） |
| G5 | 選択肢ワープ | 送信の瞬間に選択肢位置が入れ替わる |
| G6 | 全崩壊同時発動 | 20崩壊が常時・同時に動作 |
| G7 | スコア煽り | 終盤で「スコア:0」表示→実スコア表示の嘘 |
| G8 | 易問トラップ | 「1+1=?」の選択肢が「田/毎/由/甲」など紛らわしい漢字 |

### クリア条件
- 20問中20問正解（ノーミス）
- これでSランク獲得

## 6. 問題データベース

### 6-1. 総問題数
**1000問**

### 6-2. ジャンル配分

| ジャンル | 比率 | 問題数 |
|---|---|---|
| 算数 | 20% | 200問 |
| 英語 | 20% | 200問 |
| 国語（漢字・ことわざ・文法・文学） | 20% | 200問 |
| 理科 | 10% | 100問 |
| 社会 | 10% | 100問 |
| その他（音楽・美術・家庭科・体育） | 20% | 200問 |
| **合計** | **100%** | **1000問** |

### 6-3. 難易度配分（各ジャンル内）
- レベル1（基礎）: 30%
- レベル2（小6標準）: 40%
- レベル3（忘れてそう系）: 30%

### 6-4. 「忘れてそう系」方針
共感バズの核。大人が見て「これ習ったっけ…」となる単語を狙う。

**例：**
- 理科: 維管束、蒸散、胚珠、師管、木部、気孔
- 社会: 班田収授法、勘合貿易、樺太千島交換条約、墾田永年私財法
- 国語: 呉音と漢音、助動詞の活用、『つれづれなるままに』作者
- 家庭科: まつり縫い、玉結び、5大栄養素
- 音楽: 八分音符の長さ、ト音記号の正式名、変ロ長調

### 6-5. 画像使用ルール
**画像素材は使用禁止。全問題をコード描画・テキストのみで表現する。**

理由: 画像管理の工数とバグリスクを排除するため。

- OK例: 数式、漢字、化学式、音楽記号（♩♪）、簡易図形（△□○）
- NG例: 地図問題、絵画判別、楽譜読み取り、実験器具イラスト

### 6-6. データ形式
```json
{
  "id": "SC_0001",
  "genre": "理科",
  "subgenre": "植物",
  "difficulty": 3,
  "question": "植物の茎の中で、水や養分の通り道になる束状の組織を何と呼ぶか？",
  "choices": ["維管束", "気孔", "胚珠", "師管"],
  "answer": 0,
  "explanation": "維管束は道管と師管を含む組織。"
}
```

### 6-7. ファイル分割
ジャンルごとにJSON分割し、初回ロード負荷を下げる。

```
/data/questions/
  ├ math.json       (200問)
  ├ english.json    (200問)
  ├ japanese.json   (200問)
  ├ science.json    (100問)
  ├ social.json     (100問)
  └ others.json     (200問)
```

### 6-8. 生成方法
Claudeに100問単位のバッチで生成依頼。
- 各バッチで難易度分布を保つ
- 生成後に重複チェック・校正フェーズを設ける

## 7. 崩壊UI

### 7-1. 採用候補リスト（選別前）

**【小物】CSS/transform中心・軽量**
1. 反転タップ
2. 文字盤見えない（記憶打ち）
3. 文字盤あべこべ
4. 解答欄見えない
5. 問題徐々にしか見えない
6. 画面回転
7. ズーム暴走
8. ミラー
9. 色覚破壊（同系色）
10. グリッチ
11. フェードアウト（問題文消失）
12. 選択肢シャッフル
13. キーボード縮小
14. 背景同化
15. スクロール強制
16. ゴースト入力
17. カーソル暴走
18. 点滅
19. ぼかし
20. フォント極小
21. 余白暴走

**【中物】ロジック軽め**
22. 入力遅延
23. タップ無反応（フリック可）
24. 文字順逆転（入力が逆順で並ぶ）
25. 問題文逆順表示
26. ダミー選択肢（1文字違い）
27. 意味のない高速カウントダウン（飾り）
28. 問題ランダム差し替え
29. 選択肢文字の1秒ごと変化
30. 入力が1文字ごと消える
31. キーボード配置ランダム化
32. 決定ボタン移動
33. スクロール逆方向
34. ミスタップ感度超高
35. 偽エラー表示
36. 時計逆行演出

**【中物悪質】認知負荷系**
37. 選択肢が問題文に紛れ込む
38. 正解フィードバック嘘
39. 進捗バー嘘
40. ヒント妨害
41. 前問引き継ぎ
42. 正解音がプレッシャー音
43. 問題番号偽装
44. 同じ問題連続
45. 他プレイヤーのスコア偽表示

**【大物】実装重め**
46. 画面分割
47. タイピング音遅延
48. 前問の正解率で次の難度変動
49. 即死演出の多層化

**【却下】バグリスク高・実装重**
- ~~ダブルタップ必須~~（意味不明）
- ~~時間系ペナルティ~~（意図に反する）
- ~~音声干渉（文字ダミー/音声正解）~~（実装検討継続）
- ~~問題文が画像化~~（画像使用禁止方針）
- ~~手書き入力必須~~（OCR重い）
- ~~シェイク検出~~（バグリスク）
- ~~ジャイロ操作~~（バグリスク）
- ~~顔認識エラー演出~~（カメラ権限で離脱）
- ~~マルチタスク強制広告~~（実装重）
- ~~自分の名前を選択肢~~（名前入力フロー必要）
- ~~OS通知攻撃~~（本物通知との混同リスク）

### 7-2. 最終選別方針
UIパターン実装フェーズで下記基準で再選別：
- 実装時間: 1機能あたり2時間以内を目安
- バグ発生リスク: 低のみ採用
- 端末差異: iOS/Android両方で同挙動のもののみ

### 7-3. ステージごとの発動ロジック
- 各ステージで使用する崩壊UIは、上記リストからランダム抽選
- 同一ステージでも毎プレイで発動組み合わせが変わる（リプレイ性）
- Stage9はパラメータMAX、Stage10は全崩壊同時発動

## 8. リザルト画面（バズ装置）

### 8-1. 表示項目
```
┌─────────────────────────┐
│   ステージクリア！       │
├─────────────────────────┤
│   ランク：  A           │
│   ★★★★☆              │
│                         │
│   タイム：   04:23      │
│   スコア：   1,840 pt   │
│   正解数：   17/20      │
│                         │
│   あなたは上位 3.2%     │
│   ≒ 早慶合格率 同等     │
│                         │
│   [シェア]  [もう一度]  │
└─────────────────────────┘
```

### 8-2. ランク判定

| ランク | 条件 | 対応% |
|---|---|---|
| S | Stage10完走（20/20） | 0.01% |
| A | Stage9クリア | 1% |
| B | Stage7クリア | 10% |
| C | Stage5クリア | 30% |
| D | Stage3クリア | 60% |
| E | Stage1クリア | 90% |
| F | Stage1未達 | 下位 |

### 8-3. スコア計算式
```
score = 正解数 × 100
      + タイムボーナス（max 500、早いほど高い）
      + ノーミスボーナス（20連続正解で +500）
      + Stage到達ボーナス（Stage n到達で n×100）
      + 崩壊UI数ボーナス（クリア時の崩壊数×50）
```

### 8-4. 上位%と現実換算ラベル

**上位バズ（自慢スクショ用）**

| 上位 | ラベル候補（ランダム表示） |
|---|---|
| 0.01%以下 | 宇宙飛行士選抜通過、ノーベル賞候補、プロ棋士レベル |
| 0.1% | 東大理III合格率、オリンピック出場者 |
| 1% | 医師、年収2000万円、東大生 |
| 3% | 早慶合格率、TOEIC 900超 |
| 5% | MARCH合格、年収1000万円 |
| 10% | 公務員合格、大卒上位 |

**下位バズ（ネタスクショ用）**

| 下位 | 辛辣ラベル候補 |
|---|---|
| 30% | 凡人、普通人 |
| 50% | 人類の中央値 |
| 70% | 小学生に負けるレベル |
| 90% | ぴえん、小学校やり直し推奨 |
| 95%以下 | 伝説のアホ、令和の維管束知らず、無免許運転レベル |
| Fランク | 称号「現代の縄文人」 |

### 8-5. 下位ラベル設計方針
下位の人も「これはこれで面白い」と感じて**スクショしたくなる**設計。
辛辣だが悪意は感じさせない、ネット文化的な笑いに寄せる。

## 9. シェア機能

### 9-1. 目的
**SNS拡散の主動脈**。リザルト画像化→ネイティブシェアシート。

### 9-2. 実装
- html2canvasでリザルト画面をPNG化
- @capacitor/shareで標準シェアシート起動
- 画像にゲーム名・URL・ハッシュタグ埋め込み

### 9-3. シェア画像に含める要素
- ゲームタイトル
- ランク（でかく表示）
- スコア
- 上位%ラベル
- QRコード or App StoreリンクのURL
- ハッシュタグ（#クソゲー #維管束 等）※タイトル確定後に決定

## 13. リスク管理

### 13-1. 技術リスク
| リスク | 対策 |
|---|---|
| WebViewのiOS/Android挙動差 | 両OS実機で全崩壊UIを検証 |
| html2canvasが特定UIで破綻 | シェア画像は別途シンプル構成で生成 |
| 起動時の問題DBロード遅延 | ジャンル別遅延ロード |

### 13-2. 審査リスク
| リスク | 対策 |
|---|---|
| Apple審査：機能不十分で却下 | ゲームとして完結させる、課金なし |
| Apple審査：誇大広告 | ストア説明文は事実ベースに |
| プライバシーマニフェスト不備 | Capacitor 8の公式ガイド準拠 |

### 13-3. バズ失敗リスク
| リスク | 対策 |
|---|---|
| 告知力不足 | 開発中からX/TikTokで開発過程を発信 |
| Sランク不可能すぎて話題化しない | Stage10は理論上クリア可能な調整に留める |
| 共感が取れない | 「維管束」系の忘却問題を十分に混ぜる |
